# 求职分析智能体设计方案

## 1. 目标与设计原则

### 目标

构建一个轻量化的求职分析智能体，满足以下核心能力：

1. 支持上传 Word 或 PDF 格式简历并完成结构化解析
2. 支持上传岗位 JD 文本或文件并完成结构化分析
3. 基于简历与岗位要求的匹配结果，输出具体、可执行的简历优化建议
4. 后端使用 Python，智能体编排使用 LangGraph，整体方案保持轻量、易落地、易扩展

### 设计原则

- **轻量优先**：优先采用 FastAPI + LangGraph + Pydantic 的最小可行架构
- **闭环优先**：上传、解析、匹配、建议生成必须形成完整链路
- **结构化优先**：所有关键输出都要有明确 Schema，避免自由文本失控
- **鲁棒优先**：考虑空文件、扫描 PDF、解析失败、模型输出异常等真实场景
- **可扩展优先**：后续可平滑接入多模型、OCR、多轮修改、评分体系

## 2. 优化后的整体架构

### 分层架构

```text
┌────────────────────────────────────────────────────────────────────┐
│                           API Layer                               │
│                FastAPI / Upload / Validation / Auth               │
├────────────────────────────────────────────────────────────────────┤
│                         Service Layer                             │
│      文件保存  |  文本提取  |  结构化抽取  |  分析聚合  |  输出组装   │
├────────────────────────────────────────────────────────────────────┤
│                        LangGraph Layer                            │
│ parse_resume ─┐                                                    │
│               ├─> analyze_match -> generate_suggestions -> format │
│ parse_job   ──┘                                                    │
├────────────────────────────────────────────────────────────────────┤
│                      Infrastructure Layer                         │
│ LLM Client | Pydantic Schema | DOCX/PDF Parser | OCR(Optional)   │
└────────────────────────────────────────────────────────────────────┘
```

### 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| Web 框架 | FastAPI | 异步接口、开发效率高 |
| 智能体编排 | LangGraph | 适合状态流转、节点编排、后续多轮扩展 |
| 数据模型 | Pydantic v2 | 结构化校验、便于约束模型输出 |
| 文档解析 | python-docx, pypdf | 处理 docx 与文本型 PDF |
| OCR 兜底 | rapidocr-onnxruntime 或 paddleocr（可选） | 扫描版 PDF/图片简历可选支持 |
| 模型接入 | langchain-openai / 自定义 provider | 兼容 OpenAI 或国产模型 |
| 部署 | Uvicorn | 轻量部署 |
| 日志 | loguru 或 logging | 记录 trace_id、异常栈、模型耗时 |

### 为什么仍然选择 LangGraph

如果只是“串行执行 3 个函数”，LangGraph 的价值并不明显；但本项目后续很容易扩展出如下能力：

- 简历解析与 JD 解析并行执行
- 技能匹配、经验匹配、关键词匹配并行执行
- 失败节点重试或降级
- 多轮“继续优化这一段简历”的对话式修改
- 持久化 checkpoint，支持长流程恢复

因此本项目保留 LangGraph，但只使用最必要的状态编排能力，不引入过重依赖。

## 3. 核心流程设计

### 端到端链路

```text
上传简历/岗位文件
        ->
文件校验与保存
        ->
文本提取与清洗
        ->
LLM 结构化抽取
        ->
匹配分析
        ->
生成修改建议
        ->
返回结果
```

### 推荐执行流程

1. 用户上传简历文件
2. 用户提供岗位文本或岗位文件
3. 服务端完成文件格式校验、大小限制、临时文件保存
4. 简历与岗位分别提取文本
5. 对原始文本进行清洗、分段、去噪
6. 使用 LLM + Pydantic Schema 输出结构化数据
7. 基于结构化结果执行技能、经验、关键词匹配
8. 汇总差距点并生成高优先级修改建议
9. 返回匹配结果、问题列表、建议示例

### 关键优化点

- 简历与岗位解析必须都支持文件输入，不能只支持 JD 文本
- LLM 输出不能只要求“返回 JSON”，必须走 Schema 校验
- 建议生成前必须保留证据链，否则容易输出空泛建议
- 工作流中适合并行的节点尽量并行，减少总耗时

## 5. 文件解析与结构化抽取设计

### 5.1 统一解析链路

简历与岗位文件都采用统一三段式：

```text
extract_text -> clean_text -> structured_extract
```

这样可以避免“简历有解析器、JD 只有提示词”的不一致设计。

### 5.2 文本提取器

```python
# app/parsers/document_parser.py
from pathlib import Path
import docx
from pypdf import PdfReader


class DocumentParser:
    def extract_text(self, file_path: str) -> str:
        suffix = Path(file_path).suffix.lower()
        if suffix == ".docx":
            return self._parse_docx(file_path)
        if suffix == ".pdf":
            return self._parse_pdf(file_path)
        raise ValueError("unsupported file type")

    def _parse_docx(self, file_path: str) -> str:
        document = docx.Document(file_path)
        return "\n".join(p.text.strip() for p in document.paragraphs if p.text.strip())

    def _parse_pdf(self, file_path: str) -> str:
        reader = PdfReader(file_path)
        texts = [page.extract_text() or "" for page in reader.pages]
        return "\n".join(t.strip() for t in texts if t.strip())
```

### 5.3 文本清洗器

```python
# app/parsers/text_cleaner.py
import re


def clean_text(text: str) -> str:
    text = text.replace("\u00a0", " ")
    text = re.sub(r"\n{3,}", "\n\n", text)
    text = re.sub(r"[ \t]{2,}", " ", text)
    return text.strip()
```

### 5.4 结构化抽取器

推荐使用“模型结构化输出 + Pydantic 校验 + 失败重试”。

```python
# app/extractors/resume_extractor.py
from app.models.resume import ResumeData


class ResumeExtractor:
    def __init__(self, llm_client):
        self.llm_client = llm_client

    def extract(self, text: str) -> ResumeData:
        prompt = f"""
你是一名简历信息抽取助手。
请从以下简历文本中抽取结构化信息。
如果字段缺失，返回 null 或空列表，不要编造。

简历文本:
{text}
"""
        return self.llm_client.invoke_structured(
            prompt=prompt,
            schema=ResumeData,
        )
```

岗位 JD 结构化抽取同理，输入可以是文本，也可以先从文件中提取文本后再调用。

### 5.5 解析失败的兜底策略

- 文本为空：直接返回“文件无法解析出有效文本”
- PDF 加密或损坏：返回明确错误码
- 扫描版 PDF：尝试 OCR，可配置为可选能力
- 结构化输出失败：自动重试 1 到 2 次
- 多次失败：降级返回原始文本摘要，避免接口完全不可用

## 6. LangGraph 工作流设计

### 6.1 状态定义

优化后的状态分为三层：输入层、中间结果层、输出层。

```python
# app/workflow/state.py
from typing import Annotated, Optional, TypedDict
from langgraph.graph.message import add_messages
from app.models.resume import ResumeData
from app.models.job_requirement import JobRequirement
from app.models.match_result import MatchSummary
from app.models.suggestion import ResumeSuggestion


class AgentState(TypedDict, total=False):
    # 原始输入
    resume_file_path: str
    job_file_path: Optional[str]
    job_text: Optional[str]

    # 文本层
    resume_raw_text: str
    job_raw_text: str

    # 结构化结果
    resume_data: ResumeData
    job_requirement: JobRequirement

    # 分析结果
    match_summary: MatchSummary

    # 输出结果
    suggestions: list[ResumeSuggestion]
    final_response: dict

    # 运行元信息
    trace_id: str
    error_message: Optional[str]
    messages: Annotated[list, add_messages]
```

### 状态设计说明

- 输入文件路径与结构化结果分离，解决“未解析前状态不成立”的问题
- `job_text` 和 `job_file_path` 二选一，兼容文本输入与文件输入
- `final_response` 用于统一封装接口输出
- `error_message` 用于工作流失败节点降级

### 6.2 工作流拓扑

```python
# app/workflow/graph.py
from langgraph.graph import START, END, StateGraph
from app.workflow.state import AgentState
from app.workflow.nodes import (
    parse_resume_node,
    parse_job_node,
    analyze_match_node,
    generate_suggestions_node,
    format_response_node,
)


def build_workflow():
    graph = StateGraph(AgentState)

    graph.add_node("parse_resume", parse_resume_node)
    graph.add_node("parse_job", parse_job_node)
    graph.add_node("analyze_match", analyze_match_node)
    graph.add_node("generate_suggestions", generate_suggestions_node)
    graph.add_node("format_response", format_response_node)

    graph.add_edge(START, "parse_resume")
    graph.add_edge(START, "parse_job")
    graph.add_edge("parse_resume", "analyze_match")
    graph.add_edge("parse_job", "analyze_match")
    graph.add_edge("analyze_match", "generate_suggestions")
    graph.add_edge("generate_suggestions", "format_response")
    graph.add_edge("format_response", END)

    return graph.compile()
```

### 6.3 节点职责拆分

| 节点 | 输入 | 输出 | 说明 |
|------|------|------|------|
| parse_resume | 简历文件路径 | `resume_raw_text`, `resume_data` | 提取简历文本并结构化 |
| parse_job | 岗位文本或文件路径 | `job_raw_text`, `job_requirement` | 提取 JD 文本并结构化 |
| analyze_match | `resume_data`, `job_requirement` | `match_summary` | 计算技能、经验、关键词匹配 |
| generate_suggestions | `match_summary` 等 | `suggestions` | 输出具体简历优化建议 |
| format_response | 所有结果 | `final_response` | 封装统一接口响应 |

### 6.4 为什么这里不拆太多节点

为了保持轻量化，第一版不建议把技能匹配、经验匹配、关键词匹配拆成过多子图。推荐做法：

- **MVP**：一个 `analyze_match_node` 统一处理
- **v2**：拆成并行节点，进一步提升可维护性与可观测性

## 7. 匹配分析设计

### 分析维度

1. **硬技能匹配**：如 Python、LangGraph、FastAPI、SQL
2. **经验匹配**：如 3 年以上开发经验、AI 项目经验、后端经验
3. **关键词覆盖**：判断简历是否覆盖 JD 高频关键词
4. **学历匹配**：是否满足学历、专业、加分项要求
5. **硬性阻塞项**：如必须掌握某技术但简历完全未体现

### 建议的分析输出

```python
{
    "overall_score": 78.0,
    "skill_gap": {
        "matched": ["Python", "FastAPI"],
        "missing": ["LangGraph", "Prompt Engineering"],
        "weak": ["LLM 应用经验"]
    },
    "experience_match": {
        "matched_experience": ["后端开发", "接口设计"],
        "missing_experience": ["求职产品业务经验"],
        "score": 72.0
    },
    "keyword_match": {
        "matched_keywords": ["异步接口", "简历解析"],
        "missing_keywords": ["多智能体", "结构化输出"]
    },
    "hard_blockers": [
        "岗位要求 LangGraph 经验，但简历中未体现"
    ]
}
```

### 分析策略建议

- 第一层采用规则匹配，保证稳定性
- 第二层可结合 LLM 做语义归一化，如把“熟悉 Python Web 开发”映射到 `FastAPI/Flask`
- 不建议一开始就引入向量数据库，当前场景规则 + LLM 已足够

## 8. 建议生成设计

### 建议必须满足的要求

- 每条建议都对应一个明确问题
- 每条建议都要关联岗位要求或匹配缺口
- 每条建议都尽量给出修改前后示例
- 建议分优先级，避免输出十几条价值接近的内容

### 建议生成提示词要点

```python
def generate_suggestions_prompt(match_summary, resume_data, job_requirement) -> str:
    return f"""
你是一名资深求职顾问，请根据岗位要求和简历差距生成简历修改建议。

请遵守以下要求：
1. 只输出具体、可执行的建议
2. 优先处理硬性要求缺口
3. 每条建议包含问题、原因、建议、修改示例
4. 如果简历已有相关内容，优先建议增强表达与量化结果
5. 不要编造候选人没有的经历

岗位要求:
{job_requirement}

简历信息:
{resume_data}

匹配分析:
{match_summary}
"""
```

### 推荐输出样式

1. **必须修改**
   - 补充 LangGraph 相关项目经历或技能描述
   - 将模糊描述改为量化成果

2. **建议修改**
   - 在项目经历中补充与岗位关键词更一致的表述
   - 调整技能模块顺序，先展示岗位强相关技能

3. **可选优化**
   - 增加个人总结，突出求职方向
   - 统一日期格式和模块标题格式

    # 2. 保存上传文件
    # 3. 调用 analyzer service
    # 4. 返回结构化结果

### 目录优化说明

- `parsers` 只负责文本提取与清洗
- `extractors` 只负责 LLM 结构化抽取
- `services` 做业务聚合，不把复杂逻辑塞进路由
- `workflow` 只做状态编排
- `core` 放配置、异常、日志，避免散落

## 11. 轻量化实现建议

### 必选能力

- FastAPI + LangGraph + Pydantic
- docx/pdf 文本提取
- 单模型 provider 封装
- 匹配分析与建议生成
- 基础异常处理与日志

### 可选能力

- OCR 兜底
- 多模型切换
- Redis / 数据库存储任务结果
- 向量检索
- 多轮对话式简历优化

### 第一版不要做太重的部分

- 不急着上向量数据库
- 不急着做账号系统
- 不急着拆多服务
- 不急着做前端复杂编排

先保证“上传 -> 解析 -> 匹配 -> 建议”这条主链路稳定可用。

## 12. 工程风险与对策

| 风险 | 表现 | 对策 |
|------|------|------|
| PDF 难解析 | 文本缺失、顺序混乱 | 优先支持文本型 PDF，扫描版走 OCR 兜底 |
| LLM 输出不稳定 | JSON 非法、字段缺失 | 使用 Schema 校验 + 重试 |
| 建议过于空泛 | 缺少落地性 | 增加问题字段、证据链、前后示例 |
| 响应耗时高 | 多次调用模型 | 并行解析、减少重复 prompt |
| 用户输入脏数据 | 空文件、格式错 | 文件校验、明确错误码 |

## 15. 方案总结

这版优化后的方案相较原方案，重点解决了以下问题：

1. **输入链路闭环**：岗位既支持文本，也支持 Word/PDF 文件
2. **状态设计合理**：输入、解析结果、分析结果、输出结果彼此分层
3. **LLM 输出可控**：引入 Pydantic Schema 校验与失败重试
4. **LangGraph 用法更合适**：保留并行与扩展能力，但不过度设计
5. **建议更可执行**：每条建议都绑定问题、原因、示例和岗位要求
6. **工程上更稳**：补充了异常处理、错误码、日志、目录分层

整体来看，这是一套适合快速落地的 Python + LangGraph 轻量化求职分析智能体方案。第一版即可支持核心业务闭环，后续也能逐步扩展为更完整的求职辅助系统。
