### 1.4 当前工作流图(LangGraph,v0.3)

```text
                  ┌─→ parse_resume ─┐
  START ──────────┤                 ├─→ dispatch_matchers (fan-in,空节点)
                  └─→ parse_job  ──┘                │
                                                     │ Send fan-out
                                                     ▼
                              skill_gap ┐
                              experience├──→ aggregate_report ─→ generate_suggestions → END
                              keywords  ┘
                              (3 个并行)
```

每个节点都套 `_with_stage_tracking` 装饰器,统一接入 progress 系统:

```python
# app/workflow/graph.py
def _with_stage_tracking(node_func):
    def wrapper(state):
        stage = stage_by_node(node_func.__name__)
        set_current_stage(stage)
        t0 = time.monotonic()
        result = node_func(state)
        logger.info(f"节点 {node_func.__name__} 完成,耗时 {(time.monotonic()-t0)*1000:.0f}ms")
        return result
    return wrapper
```

- **并行抽取**:`parse_resume` / `parse_job` 都从 `START` 扇出,互不依赖,节省 1 次 LLM 串行耗时
- **fan-in 汇聚点**:`dispatch_matchers` 是个空节点,避免两条扇入边触发 LangGraph 的 `InvalidUpdateError`
- **并行匹配**:`Send` 模式把状态同时推给三个匹配节点,谁先算完谁先返回,总耗时取最长者
- **fan-in 汇总**:`aggregate_report` 等待三个匹配节点全部完成,内部调 `LLMClient.chat_json(schema=LLMScoreOutput)` 计算综合评分,`field_validator` 钳制到 [0,100]
- **降级兜底**:`generate_suggestions` 双层 fallback(LLM 失败 → 规则建议),保证建议总能产出
- **同步 ainvoke / 流式 astream**:两套入口复用同一张图,流式用 `asyncio.Queue + call_soon_threadsafe` 串联节点事件与 LLM token 进度

### 2.2 LLM 客户端统一封装(v0.3)

[app/extractors/llm_client.py](file:///c:/Users/10408/Desktop/job/app/extractors/llm_client.py) 是唯一和 LangChain / OpenAI 兼容协议打交道的入口。

#### 公开 API

| 方法 | 签名 | 用途 |
|---|---|---|
| `chat_json` | `(system, user, schema, *, max_retries=1, progress_callback=None)` | 按 Pydantic schema 解析 JSON,失败自动重试 |
| `chat_text` | `(system, user, *, progress_callback=None)` | 自由文本输出 |
| `_invoke` | `(system, user, hint, *, progress_callback)` | 路由:有 callback 走流式,否则走同步 |
| `_invoke_stream` | `(system, user, hint, callback)` | 流式核心,逐 chunk 推送进度 |

#### 流式回调三阶段

```python
# app/extractors/llm_client.py
def _invoke_stream(self, system, user, hint, callback):
    for chunk in self._get_model().stream([...]) :
        text = chunk.content or ""
        if first_chunk:
            callback("first_token", {"chars": len(text)})   # ① 首 chunk
        elif total_chars % _STREAMING_CHARS_THRESHOLD < len(text):
            callback("streaming", {"chars": total_chars})    # ② 每 80 字符
        ...
    except Exception:
        callback("error", {"stage": ...})                    # ③ 异常
```

- 首 chunk 触发 `first_token`,前端立即转圈
- 每累计 80 字符触发 `streaming`,前端进度条平滑推进
- 异常触发 `error`,前端切到失败视图

#### JSON 解析三层容错

| 层级 | 函数 | 处理场景 |
|---|---|---|
| 1 | `_safe_json_loads` | 纯 JSON、```json``` 包裹、文本嵌套、前后解释 |
| 2 | `_coerce_payload` | LLM 直接输出 list 时自动包裹到 schema 的 List 字段 |
| 3 | `_fill_missing_fields` | 缺失字段用默认值/None 补,枚举值用包含关系模糊匹配 |

#### 回调来源优先级

`显式参数 > ContextVar > 线程安全共享变量 > None`,确保同步/流式/工作线程三种路径都能拿到。

#### provider 透明

`.env` 中 `LLM_PROVIDER=deepseek/openai` 切换 0 业务代码改动(目前仍是 `ChatOpenAI` 兼容协议,DeepSeek v4 Flash 通过 OpenAI-compatible API 接入)。

### 2.3 工作流的"双层并行"

| 阶段 | 并行方式 | 收益 |
|---|---|---|
| 抽取 | `add_edge(START, "parse_resume")` + `add_edge(START, "parse_job")` | 省 1 次 LLM 串行耗时 |
| 匹配 | `Send` fan-out 到 3 个节点 | 总耗时取最长者,3× → 1× |
| 建议生成 | 串行 | LLM 1 次,无法并行 |

### 2.4 进度推送(流式核心,v0.3)

[app/workflow/progress.py](file:///c:/Users/10408/Desktop/job/app/workflow/progress.py) 定义了**前后端共用的单一事实源** `STAGES`,共 8 个阶段,每个阶段有固定 percent 区间:

```text
upload 0-5  | parse_resume 5-20 | parse_job 20-35
skill_gap 35-65 | experience 65-78 | keywords 78-88
aggregate 88-92 | suggestions 92-99
```

#### 推送链路

```text
LLMClient._invoke_stream(...)
   │ chunk / first_token / streaming(每 80 字符)
   ▼
progress_callback(phase, {chars})      [在 LangGraph 工作线程中]
   │
   ▼ loop.call_soon_threadsafe
asyncio.Queue.put_nowait(...)
   │
   ▼ 主生成器消费
yield NDJSON: {type: "progress", percent, message, chars}
```

#### 阶段追踪 — Lock + 共享变量(v0.3 改造)

v0.2 用 ContextVar,在 LangGraph `astream` 跨线程场景下不可靠。v0.3 改为 `threading.Lock + 模块级变量`:

```python
# app/workflow/progress.py
_lock = threading.Lock()
_progress_shared: dict | None = None
_stage_shared: StageDef | None = None

def set_progress_callback(cb):
    global _progress_shared
    with _lock:
        _progress_shared = cb

def get_progress_callback():
    with _lock:
        return _progress_shared
```

`StageDef` 数据类:

```python
@dataclass(frozen=True)
class StageDef:
    key: str                # "parse_resume"
    label: str              # "解析简历"
    node: str               # LangGraph 节点名
    start_percent: float    # 5.0
    end_percent: float      # 20.0
    is_llm: bool = True     # 是否 LLM 调用阶段
```

`compute_streaming_percent(stage, phase, chars)` 把字符数线性映射到 `[start_percent, end_percent]` 区间。

#### 节点装饰器统一接入 stage(v0.3)

```python
# app/workflow/graph.py
def _with_stage_tracking(node_func):
    def wrapper(state):
        stage = stage_by_node(node_func.__name__)
        set_current_stage(stage)
        t0 = time.monotonic()
        result = node_func(state)
        logger.info(f"节点 {node_func.__name__} 完成,耗时 {(time.monotonic()-t0)*1000:.0f}ms")
        return result
    return wrapper
```

所有节点注册时统一套用,LangGraph 节点与 progress 系统的 stage 自动打通,无需每个节点手动 `set_current_stage`。

#### NDJSON 事件流

```typescript
type StreamEvent =
  | { type: 'meta',          trace_id: string }
  | { type: 'stage_start',   index: number, label: string, key: string }
  | { type: 'progress',      percent: number, message: string, chars: number }
  | { type: 'stage_end',     index: number, key: string }
  | { type: 'done',          data: AnalysisResult, trace_id: string }
  | { type: 'error',         stage: string, code: string, message: string };
```

每行一个 JSON 对象,以 `\n` 分隔。前端严格按 `type` 字段分发,不允许依赖顺序。

#### 异常透传

- 节点抛 `AppError` → 推送 `{"type":"error","stage":...,"code":...,"message":...}`
- 节点抛其他异常 → 推送 `{"type":"error","stage":...,"code":"workflow_error","message":...}`
- langgraph 0.2.x 的 `astream` 不发 `__end__`,因此用累积的 `initial` 作为 `final_state`
- 流式 fetch 失败 → 前端 `try/fallback` 调 `POST /api/v1/analyze` 同步兜底

### 2.5 匹配设计(v0.3)

#### 三层匹配器入口

| 匹配器 | 实现 | LLM 依赖 | 失败回退 |
|---|---|---|---|
| 技能 | `_semantic_skill_gap` + `_literal_skill_gap` | 1 次 LLM | 切字面匹配 |
| 经验 | 规则 + 年限计算 | 无 | — |
| 关键词 | 规则(子串归一化) | 无 | — |
| **综合评分** | `_llm_score`(`aggregate_report_node` 内部) | 1 次 LLM | 抛 `WorkflowError`,中断工作流 |

#### 技能语义匹配关键点

- **单次 LLM 调用**:把简历原文 + 硬/软技能列表一并塞进 prompt,一次返回所有判断
- **等价物参考**:`PgVector ≡ Milvus / Qdrant`、`FastAPI(asyncio) ≈ aiohttp / Sanic`、`SQLAlchemy ≈ SQLModel` 等
- **括号示例不构成硬要求**:`LLM (OpenAI / DeepSeek) 等至少一种` 只需要命中 LangGraph/LangChain 就算 matched
- **三级容错查找**:精确 → 归一化(`re.sub(r"[\s/(),]+", "", s)`) → 短文本包含
- **Pydantic 强约束**:`_SkillEntry` + `_SkillGapSchema` 严格约束 LLM 输出

#### 综合评分 LLM 输出 Schema(v0.3 新增)

```python
# app/workflow/nodes.py
class LLMScoreOutput(BaseModel):
    """LLM 综合评分输出."""
    overall_score: float = Field(..., description="0-100 综合匹配分")
    score_reasoning: str = Field(..., description="评分理由(2-4 句话)")
    strengths: List[str] = Field(default_factory=list)
    weaknesses: List[str] = Field(default_factory=list)

    @field_validator("overall_score")
    @classmethod
    def clamp_score(cls, v: float) -> float:
        """强制钳制到 [0, 100] 区间,保留 1 位小数."""
        return round(max(0.0, min(100.0, v)), 1)
```

#### 5 档评分锚点(v0.3 引入)

`aggregate_report_node` 调 LLM 时注入 `_SCORE_SYSTEM_PROMPT`,给出 5 档可量化锚点:

| 区间 | 含义 |
|---|---|
| **90+** | 几乎完美匹配,核心技能全覆盖,经验充足 |
| **75-89** | 整体匹配,大部分核心技能覆盖,有少量可补足缺口 |
| **60-74** | 部分匹配,存在若干技能或经验缺口 |
| **40-59** | 匹配度较低,核心技能缺失较多或经验明显不足 |
| **<40** | 严重不匹配 |

约束要点:
1. `overall_score` 综合技能 / 经验 / 关键词三维度,**不能简单取平均**
2. 核心技能缺失扣分**重于**边缘技能缺失
3. 经验不足但技能覆盖好 → 高于 技能缺失但经验充足
4. 关键词覆盖权重**低于**技能和经验
5. `score_reasoning` 必须引用具体证据

#### 安全修复:v0.2 → v0.3

**v0.2 用 `eval()` 解析 LLM 输出** —— 当 LLM 倾向输出 Python 字面量(常见于跨类目简历)如 `List[Dict[str, str]]`,触发 `NameError: name 'List' is not defined`,影响 3.3% 用例。**v0.3 彻底移除 `eval`,改用 `LLMScoreOutput` Pydantic Schema + `_safe_json_loads` / `_coerce_payload` / `_fill_missing_fields` 三层容错**(详见 §2.2)。

### 2.6 降级策略(v0.3)

#### 降级矩阵

| 失败点 | 降级方案 | 影响范围 |
|---|---|---|
| 文档解析失败 | 返回明确错误码,API 422 | 整次请求失败 |
| **LLM JSON 解析失败** | 重试 1 次 → `_safe_json_loads` / `_coerce_payload` / `_fill_missing_fields` 三层容错 → `ExtractError` 抛出 | 节点失败 |
| 技能语义匹配失败 | 自动切到字面匹配(同 API) | 仅技能维度 |
| **综合评分 LLM 失败** | 抛 `WorkflowError`,工作流终止,前端收到 `error` 事件 | **v0.3 设计上不降级**,分数是最重要的契约 |
| **建议生成 LLM 失败(模块层)** | `_fallback_suggestions` 启发式:missing 技能 → HIGH / 经验不足 → HIGH / 关键词缺失 → MEDIUM | **第一层 fallback**(suggestion_generator.py 内部) |
| **建议生成 LLM 失败(节点层)** | 节点 `try/except` 兜底,显式调 `_fallback_suggestions` | **第二层 fallback**(nodes.py 节点) |
| LLM key 缺失 | 启动时仍允许启动,首次请求 502 | 整次请求失败 |
| 数据库不可用 | 缓存写入失败静默降级,主分析不受影响;面试题预测返回 503 | 面试题功能不可用 |

#### 建议生成双层 fallback(v0.3 强化)

v0.2 只在节点层兜底。v0.3 增加模块层兜底,允许 `analyzer` 直接调 `generate_suggestions` 时也安全:

```python
# app/workflow/nodes.py · 节点层(外层)
def generate_suggestions_node(state):
    try:
        suggestions = generate_suggestions(resume, job, report, LLMClient())
    except Exception as exc:
        logger.exception("generate_suggestions 失败,降级为启发式: %s", exc)
        from app.workflow.suggestion_generator import _fallback_suggestions
        suggestions = _fallback_suggestions(resume, job, report)
    state["suggestions"] = [s.model_dump() for s in suggestions]
    return state

# app/workflow/suggestion_generator.py · 模块层(内层)
def generate_suggestions(resume, job, report, llm) -> List[ResumeSuggestion]:
    try:
        result: SuggestionListOutput = llm.chat_json(
            system=_SYSTEM_PROMPT,
            user=_build_user_prompt(resume, job, report),
            schema=SuggestionListOutput,
            max_retries=2,
        )
    except Exception as exc:
        logger.warning("LLM 建议生成失败,使用回退: %s", exc)
        return _fallback_suggestions(resume, job, report)
    return _sort_suggestions(result.suggestions)
```

`_fallback_suggestions` 启发式覆盖**三类缺口**:

| 缺口类型 | 输出 | 优先级 |
|---|---|---|
| `skill_gap.missing[:3]` | "补充技能: {skill}" | HIGH |
| `experience.years_estimated < years_required` | "量化工作经历中的成果和年限" | HIGH |
| `keywords.missing[:5]` | "在简历中融入关键词: ..." | MEDIUM |

每条 fallback 建议都附 `reason` 字段说明依据,保证下游可以解释。

### 2.8 分析缓存层

#### 选型:PostgreSQL + JSONB

本项目的"缓存"本质是**持久化 + 关联查询**,而非传统临时加速层:

- **持久化需求**:分析结果需跨会话保留,Redis 的 TTL 机制与此矛盾
- **关联查询**:面试题预测需从 match_report 提取 skill_gap,从 resume_data 提取项目经历
- **JSONB 原生支持**:存储深层嵌套 Pydantic 对象,支持 GIN 索引和 JSON 路径查询
- **扩展性**:未来 RAG 知识库可直接使用 pgvector 扩展
- **async 契合**:SQLAlchemy 2.0 async + asyncpg 与 FastAPI 生态完全一致

> Redis 定位:未来如需 L1 加速层(短时间重复请求秒回),可加 Redis,与 PostgreSQL 不冲突。

#### ORM 模型

```python
class AnalysisCache(Base):
    __tablename__ = "analysis_cache"
    trace_id: Mapped[str]          # PK, UUID
    created_at: Mapped[datetime]
    resume_name: Mapped[str|None]  # 列表查询用
    job_title: Mapped[str|None]    # 列表查询用
    overall_score: Mapped[float]   # 列表查询用
    resume_data: Mapped[dict]      # JSONB,完整 Pydantic 序列化
    job_requirement: Mapped[dict]  # JSONB
    match_report: Mapped[dict]     # JSONB
    suggestions: Mapped[dict]      # JSONB
    meta: Mapped[dict]             # JSONB
```

**JSONB vs 独立列**:ResumeData、MatchReport 字段多且嵌套深,拆为独立列维护成本极高。JSONB 存完整对象,读取后 `model_validate()` 还原为 Pydantic 模型,兼顾灵活性和类型安全。

#### 写入时机

`analyzer.py` 的 `analyze()` 和 `analyze_stream()` 完成后,通过 `_persist_cache()` 异步写入:

```python
async def _persist_cache():
    async with async_session_factory() as session:
        await save_analysis(session, trace_id, ...)
```

写入失败仅记录 WARNING,不抛异常,不影响主分析结果返回。

### 2.9 面试题预测

#### 架构:独立页面 + 缓存回填

面试题预测**不修改现有工作流**,而是作为独立模块:

```text
主分析完成 → 缓存写入 PostgreSQL
                    │
                    ▼
/interview?trace_id=xxx → 从缓存读取 → 1 次 LLM → 8-12 道题
```

#### 题型分布

| 类型 | 数量 | 难度 | 来源 |
|---|---|---|---|
| technical | 4-6 | medium/hard | skill_gap.missing / partial |
| behavioral | 2-3 | medium | experience.notes |
| project_deep_dive | 2-3 | hard | resume_data.projects + skill_gap |
| situational | 0-1 | medium | job_requirement + experience |

#### 数据模型

```python
class InterviewQuestion(BaseModel):
    type: Literal["technical","behavioral","project_deep_dive","situational"]
    difficulty: Literal["easy","medium","hard"]
    question: str
    related_skill: Optional[str]
    suggested_answer_direction: str   # 3-5 个关键要点
    follow_up: Optional[str]

class InterviewPredictionOutput(BaseModel):
    questions: list[InterviewQuestion] = Field(min_length=8, max_length=12)
    summary: str
```

#### 降级策略

- 缓存不存在 → 404,前端显示"请先完成分析"
- LLM 生成失败 → 重试 1 次,仍失败返回 502
- 数据库不可用 → 主分析不受影响,面试题功能不可用

## 6. 风险与对策

| 风险 | 表现 | 对策 |
|---|---|---|
| 真实密钥泄露 | `.env` 误传 git | `.gitignore` 兜底;DeepSeek/OpenAI 控制台可轮换 |
| PDF 难解析 | 文本缺失 / 顺序乱 | 优先支持文本型 PDF;扫描版走 OCR(已留扩展位) |
| LLM 输出不稳定 | JSON 非法 / 字段缺失 | Schema 校验 + 重试 1 次 + 启发式降级 |
| 长任务前端黑盒 | 用户以为卡死 | 流式 token 级进度 + 阶段百分比 |
| 大文件上传炸内存 | OOM | 分块读取 + 累计判断 + 提前 413 |
| 双通道接口漂移 | 同步和流式行为不一致 | `analyze` 与 `analyze_stream` 共用同一个 `AgentState` 初始结构 |
| LangGraph 版本变更 | `astream` 不发 `__end__` | 用累积 `initial` 作 `final_state` |
| 工作线程拿不到 ContextVar | 进度回调丢失 | 双轨:`ContextVar + 线程安全共享变量`,优先取有效值 |
| PostgreSQL 连接异常 | 缓存写入/读取失败 | 连接池重试 + 主分析流程不依赖缓存(缓存失败不影响分析) |
| 数据库迁移失败 | 表结构不一致 | Alembic 版本管理 + 迁移前备份 |
| 测试缺位导致回归无感 | 模型替换 / prompt 改写后无法评估效果 | 三层测试 + Markdown 报告,见 §5.5 |
